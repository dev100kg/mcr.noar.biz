# 中途採用ランディングページ運用 セキュリティ整理

## 【結論】

**現行の構成と運用前提において、一般的な企業採用サイトと同等水準のセキュリティは確保できていると評価しています。**
現時点では重大な技術リスクは確認されておらず、継続的な運用管理を前提に利用可能と考えています。

なお、セキュリティはゼロリスクにはできないため、
**現実的なリスク管理を前提に継続的な確認・改善を行っていく方針です。**
追加観点があれば柔軟に取り入れますので、ご指摘ください。

## 【インフラ観点】

### サイト構成

静的サイトホスティングを採用し、攻撃面積を最小化しています。

- 静的サイトホスティング（Netlify）
  - HTML / CSS / JSのみで構成
  - サーバーサイド処理なし
  - データベースなし
  - Netlify Forms利用を今後検討
    - クラウド基盤側での保存データ暗号化（AES-256相当）

- 本社ホームページとは分離構成
  - DNS管理：Amazon Route53（**勤怠経費システムと共通**）
  - ソースコード管理：GitHub（完全に独立）
  - 作業管理：GitHub Issues / Pull Requests（完全に独立）
  - ホスティング：Netlify（完全に独立）

アプリケーション基盤（GitHub/Netlify）は分離されており、万一の障害・侵害時も影響範囲は限定しやすい構成です。会社のホームページとは、ハイパーリンク以外の直接的な接続がないため、障害やセキュリティ面でも相互影響は限定的です。

一方でDNS管理（Route53）は共通運用のため、設定変更ミスによる影響があり得ます（先日発生）。ドメイン変更時または日次で、ドメイン管理状態を出力して確認する工程を設け、再発防止に努めています。

https://github.com/dev100kg/route53check/actions

### クラウド基盤

サーバー直接公開型と比較して攻撃面積が小さい構成です。

- セキュリティ監査・認証
  - ISO27001認証取得環境
  - SOC2監査対象サービス
- CDN配信型ホスティング
  - 元サーバー非公開による侵入リスク低減
  - 分散配信によるDDoS耐性向上
  - SSL証明書管理・脆弱性対策を基盤側で継続実施

## 【運用観点】

### 現在の運用状況

- 現時点では個人情報収集は未実施
- 管理アカウントは多要素認証（MFA）導入済
  - AWS：済
  - GitHub：済
  - Netlify：済
- 管理アカウント共有なし
- 最小権限でのアクセス付与を実施
- GitHub Pull Requestベースのリリース運用
  - 承認なしで公開されない仕組み

### 今後のフォーム設置時の対応予定

通信仕様：

- HTTPS（TLS1.2以上）による通信暗号化
- Netlify基盤側での保存データ暗号化（AES-256相当）
- 常時SSL配信

想定用途：

- カジュアル面談申込フォーム
- 求人応募フォーム

導入時の対応：

- プライバシーポリシー整備
- Netlify Formsによるデータ収集
- 入力値検証・バリデーション（形式/文字数/必須項目）
- スパム対策（HoneypotやCaptcha等）を適用
- XSS / CSRF等の基本的入力対策はプラットフォーム機能と設定で実装
- 応募目的限定利用の明示
- 保存期間・削除手順を事前定義
- 添付ファイル受付なし
- 動的サーバー処理なし
- 通知メールの適切管理（リンク確認・不要データ削除）

## 【責任分界の整理】

### クラウド側責任

- インフラセキュリティ
- 通信暗号化
- データ保存暗号化
- CDN防御
- インフラ監視・パッチ管理

### 自社運用責任

- アカウント管理（MFA等）
- 応募データ取り扱い
- プライバシーポリシー整備
- アクセス権限管理
- HTTPS強制設定・公開設定・ヘッダー設定の構成管理
- DNS設定変更時の影響確認（事前/事後）
- 運用ルール遵守

## 【総合評価】

**採用LP用途として、現行構成では実用上妥当な安全水準を確保できていると評価しています。**

- インフラは大手クラウド水準
- 静的サイト構成により攻撃面積が小さい
- 本体システムとは分離構成
- 最大リスクは技術より運用管理領域
